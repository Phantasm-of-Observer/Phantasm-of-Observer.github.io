账户信息（Account Information）

在本文中，我们将研究帐户如何存储在 Substrate 中以及其底层数据结构是什么样的。 我们将看到每个帐户都与三个引用计数器相关联，以及它们如何用于管理其生命周期。 如果您的运行时存储与链上账户关联的数据，那么了解如何在您的运行时逻辑中使用这些引用计数器非常重要。

账户引用计数器（Account reference counters）

让我们更深入地了解三个引用计数器，即消费者（consumers）、提供者（providers）和充足者（sufficients）。这些计数器跟踪运行时依赖的帐户的引用。例如，当我们将数据存储在由帐户控制的哈希表下时。通过这些计数器，我们告诉Substrate不要销毁这些账户，这通常在用户试图转移他们的存续存款时触发。

  • **消费者（consumers）**指示模块依赖此帐户的次数。 使用此计数器的一个示例是在Session pallet中，当帐户在成为验证者之前设置其会话密钥时，在增加consumer之前，providers必须大于零。 见下文。

  • **提供者（providers）**指示帐户是否处于 活跃/在准备 依赖（active/ready to be depended upon）。一个使用示例是，当创建的新帐户超过现有押金时，计数器会增加。

    consumers和providers被设计为一起使用。 providers告诉 Substrate的pallet在该帐户处于活动状态之前不要存储有关该帐户的数据（即providers> 0），consumers告诉 Substrate 在所有pallet中删除有关该帐户的数据之前不要删除该帐户（即consumers == 0）。 这是为了让用户对他们存储在链上的数据负责。 如果用户想要移除他们的账户并取回现有存款，他们需要从那些链上pallet中移除依赖关系，例如清除这些pallet的链上存储数据，这会减少consumers的数目。 Pallets还具有清理功能，以减少providers以将帐户标记为在该pallet管理范围内停用。 当帐户providers达到0时，前提是consumers此时已达到 0，所有链上pallet都认为该帐户已停用。

  • **充足者（sufficients）**指示此帐户是否有任何理由可以自给自足地独立存在。使用此计数器的一个示例是在Assets pallet中，当账户有足够数量的某些资产但不拥有任何本地账户余额时。

运行时开发人员可以通过frame-system公开的inc_consumers()、dec_consumers()、inc_providers()、dec_providers()、inc_sufficients() 和dec_sufficients()更新这些计数器。 在账户生命周期中，某个计数器的每次递增调用都应该伴随着计数器的相应递减调用，否则是设计错误。

还有三个查询功能可以简化这些计数器的使用：

  • can_inc_consumer() 检查帐户是否可以使用（providers> 0）；
  • can_dec_provider() 在将providers减为 0 之前检查是否在运行时不再引用帐户（consumers == 0）；
  • is_provider_required()检查帐户是否有未完成的消费者引用（consumers> 0）。

有关详细信息，请参阅[frame-system API 文档](Pallet in frame_system::pallet - Rust (substrate.io))。

AccountData trait和实现（AccountData trait and implementation）

回到AccountInfo结构。AccountInfo中的最后一条数据是AccountData。 它可以是任何结构，只要它满足在 [frame-system::pallet::Config trait](Config in frame_system::pallet - Rust (substrate.io))中定义的关联类型 AccountData 的trait bound。 开箱即用的 Substrate 运行时将其配置为在pallet-balances中定义的 AccountData 结构，如下所示。

[frame/balances/src/lib.rs](lib.rs - source (substrate.io))

它包括：

  • 空闲余额free。未保留的余额部分。 free余额通常对大多数操作都很重要。
  • 预留余额reserved。保留余额仍属于帐户持有人，但已暂停。reserved余额可以被削减，但只有在所有free余额都被削减之后。帐户的总余额是其free余额和reserved余额的总和。
  • 冻结余额，分为misc_frozen和fee_frozen，表示free余额不能低于的余额。fee_frozen专门用于支付交易费用，而misc_frozen用于其他一切。 实际冻结余额是这两者中的最大值，它们是在账户被锁定交易时设置的。

总结（Conclusion）

到目前为止，您已经清楚地了解了帐户数据是如何存储在Substrate中的。你可以通过查看frame_system::AccountInfo API 文档和[pallet_balances::AccountData API 文档](AccountData in pallet_balances - Rust (substrate.io))来深入挖掘。
