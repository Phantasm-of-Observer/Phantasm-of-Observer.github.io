《Substrate Docs》翻译 2.1 Key Concepts - Runtime

发表于2022-01-09|更新于2022-03-04|《Substrate Docs》翻译
|阅读量:

运行时（Runtime）

区块链的运行时是定义其行为的业务逻辑。在基于Substrate的链中，运行时被称为“状态转换函数”；这是Substrate开发人员定义用于表示区块链状态的存储项以及允许区块链用户更改此状态的功能的地方。

每个Substrate节点都包含一个运行时。运行时包含链的业务逻辑。 它定义了哪些交易是有效的和无效的，并确定了链的状态如何响应交易而变化。 “外部节点”，除了运行时之外的所有东西，都不会编译为 Wasm，只能编译为原生（native）。外部节点负责处理对等节点的发现、交易池、区块和交易八卦（gossiping）、共识以及应答来自外界的 RPC 调用。 在执行这些任务时，外部节点有时需要向运行时查询信息，或者向运行时提供信息。 运行时 API 促进了外部节点和运行时之间的这种通信。

运行时APIs（Runtime APIs）

在Substrate中， sp_api crate提供了一个接口来实现运行时 API。 它旨在使开发人员能够使用impl_runtime_apis宏定义自己的自定义运行时 API。 但是，每个运行时都必须实现[Core（核心）](Core in sp_api - Rust (substrate.io))和[Metadata（元数据）](Metadata in sp_api - Rust (substrate.io))运行时 API。 除此之外，基本的 Substrate 节点还实现了以下运行时 API：

  • [BlockBuilder](BlockBuilder in sp_block_builder - Rust (substrate.io))：提供构建区块所需的功能。
  • [TaggedTransactionQueue](TaggedTransactionQueue in sp_transaction_pool::runtime_api - Rust (substrate.io))：处理验证交易队列中的交易。
  • [OffchainWorkerApi](OffchainWorkerApi in sp_offchain - Rust (substrate.io))：处理[链下功能](Off-Chain Features | Substrate_)。
  • [AuraApi](AuraApi in sp_consensus_aura - Rust (substrate.io))：使用[Aura 共识](Consensus | Substrate_)处理阻止作者身份。
  • [SessionKeys](SessionKeys in sp_session - Rust (substrate.io))：生成和解码[会话密钥](Session Keys | Substrate_)。
  • [GrandpaApi](GrandpaApi in sp_finality_grandpa - Rust (substrate.io))：将[GRANDPA](Consensus | Substrate_)确定性小工具集成到运行时。
  • [AccountNonceApi](AccountNonceApi in frame_system_rpc_runtime_api - Rust (substrate.io))：处理查询交易索引。
  • [TransactionPaymentApi](TransactionPaymentApi in pallet_transaction_payment_rpc_runtime_api - Rust (substrate.io))：处理查询交易信息。
  • [Benchmark](Benchmark in frame_benchmarking - Rust (substrate.io))：提供一种对FRAME运行时进行[基准测试](Benchmarking | Substrate_)的方法。

为了提供其定义的无分叉运行时升级功能，Substrate运行时构建为 [WebAssembly (Wasm) ](Glossary | Substrate_)字节码。 Substrate 还定义了运行时必须实现的核心原语（primitives）。

Core primitives（Core primitives）

Substrate框架对您的运行时必须向Substrate的其他层提供什么做出了最小的假设。但是有一些数据类型需要定义并且必须满足特定的接口才能在Substrate框架内工作。

它们是：

  • 哈希（Hash）：一种对某些数据的加密摘要进行编码的类型。通常只是一个 256 位的数。
  • 摘要项（DigestItem）：一种必须能够编码与共识和变更跟踪相关的许多“硬连线”替代方案之一以及与运行时中的特定模块相关的任意数量的“软编码”变体的类型。
  • 摘要（Digest）：一系列摘要项目。这将编码与轻客户端相关的所有信息，以便在区块内拥有。
  • 外来的（Extrinsic）：一种类型，表示区块链外部的单个数据，被区块链识别。这通常涉及一个或多个签名，以及某种编码指令（例如，用于转移资金所有权或调用智能合约）。
  • 区块头（Header）：代表（以加密方式或其他方式）与块相关的所有信息的类型。它包括父哈希、存储根和外部树根、摘要和块号。
  • 区块（Block）：本质上只是Header和一系列Extrinsics的组合，以及要使用的散列算法的规范。
  • 区块数（BlockNumber）：一种对任何有效区块具有的祖先总数进行编码的类型。通常为 32 位数量。

FRAME primitives（FRAME primitives）

核心 Substrate 代码库附带 FRAME，这是Parity用于Substrate运行时开发的系统，用于Kusama和Polkadot等链。FRAME定义了额外的运行时primitives，并提供了一个框架去通过组合模块（称为pallet）来轻松构建运行时。 每个pallet都封装了特定于域的逻辑，这些逻辑表示为一组存储项、事件、错误和可调度函数。FRAME开发人员可以创建自己的pallet并重复使用现有的pallet，其中包括 50 多个随Substrate提供的托盘。

还有一组额外的primitives被假定为使用Substrate FRAME构建的运行时。 这些是：

  • 调用（Call）：可以通过外部调用的调度类型。
  • 来源（Origin）：表示调用的来源。 例如，签名消息（交易）、未签名消息（固有外部）或来自运行时本身的调用（根调用）。
  • 索引（Index）：帐户索引（又名随机数）类型。 这存储了与发件人帐户关联的先前交易的数量。
  • 散列（Hashing）：运行时使用的散列算法（例如 Blake2）。
  • 账户ID（AccountId）：用于在运行时标识用户帐户的类型。
  • 事件（Event）：用于运行时发出的事件的类型。
  • 版本（Version）：表示运行时版本的类型。

尽管 FRAME 及其相关primitives可以启用许多核心运行时开发，但 FRAME 并不是开发基于 Substrate 的区块链的唯一系统。
